⚡ Port Mapping Directives

➔
📂 WORKDIR /app
Creates an isolated directory sandbox inside the container. All subsequent commands (COPY, RUN, CMD) will execute relative to this root folder context.
🔌 EXPOSE 3000
Declares to Docker that this server listens on port 3000 inside its private bridge sandbox. It acts as documentation; actual traffic requires port mapping (-p).

Mini Project: Containerize hello-api Express REST App

Deploy a real working REST API, map ports, test connections, and clean up.

$ docker build -t hello-api ./week-1/day-4/hello-api
Explanation: Compiles the lightweight node alpine blueprint. Installs dependencies inside the WORKDIR /app context.
$ docker run -d --name node-app -p 8080:3000 hello-api
Explanation: Boots the container in the background. Maps host port 8080 to the container's private exposed port 3000.
$ curl http://localhost:8080
Explanation: Sends an HTTP request. Docker redirects the packet from your computer port 8080 to container internal port 3000!
$ docker stop node-app && docker rm node-app
Explanation: Terminates container process execution and cleans up resources from host.

⚠️ Common Port Conflicts & Fixes FAQ

Q1: What happens if the Host Port (e.g. 8080) is already in use? ▼

Docker will fail to start and throw an error: port is already allocated. You can easily fix this by changing the host mapping port number (e.g., using -p 8081:3000 or -p 9000:3000) without modifying the container's code!

Q2: Can I map multiple containers to the exact same host port? ▼

No. A physical port on your computer can only bind to a single active listener process at a time. However, you can run multiple identical containers mapping to *different* host ports (e.g. Container A on -p 8080:3000 and Container B on -p 8081:3000).
