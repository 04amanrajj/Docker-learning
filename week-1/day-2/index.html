Created The container is isolated and generated from the Docker image, but has not yet started executing its main process (PID 1).

Running The main process is active and running. The container consumes CPU, RAM, and network namespaces. Stays active as long as PID 1 is alive.
3

4
Stopped The main process is terminated (sent SIGTERM, then SIGKILL). The container configuration and writeable filesystem layer remain intact, but it is inactive.
5
Deleted The container's writeable read-write container layer is destroyed and deleted. Its storage footprint is freed up. Gone completely.

Mini Project: Practice Container Lifecycle in Terminal

Now that you understand the theory, build a custom interactive **Ubuntu 24.04** container and practice executing commands inside it.

$ docker build -t ubuntu-learning ./week-1/day-2
Explanation: This reads the Dockerfile inside the week-1/day-2 directory, pulls the official lightweight Ubuntu 24.04 LTS image, installs utilities like curl and procps, and packages our custom startup script.
$ docker run -d --name ubuntu-sandbox ubuntu-learning
Explanation: Launches the container in detached (background) mode (-d) and names it ubuntu-sandbox. It will run the welcome script and remain alive in the background.
$ docker ps
Explanation: Lists active running containers. You will see ubuntu-sandbox is up, displaying its container ID and names.
$ docker exec -it ubuntu-sandbox bash
Explanation: Opens a secure, interactive command shell (bash) inside your running Ubuntu container. The -it flags keep the shell input/output interactive. Run commands like apt-get update, top, or hostname once inside!
$ docker stop ubuntu-sandbox
Explanation: Shuts down the container by terminating its main process. The container will move to the Stopped (Exited) state.
$ docker rm ubuntu-sandbox
Explanation: Deletes the container and unbinds its resource layers. The container will be moved to the Deleted (Offline) state.

Quick Knowledge Check

Q1: What happens to files inside a container when you run 'docker stop'? ▼

The container filesystem state remains fully preserved on disk. When you run docker start again, the container boots up with all your files and changes exactly where you left them!

Q2: What is the main difference between 'docker stop' and 'docker pause'? ▼

docker stop shuts down the container's processes and terminates memory execution. docker pause suspends the CPU execution using cgroups freezer, keeping the container's exact RAM state intact but halting processor cycles entirely.
