CLI Environmental Variables Walkthrough

Compile the Node image. The Dockerfile predefines built-in default values using the ENV instruction so the application is immediately ready to run out of the box.

$ docker build -t env-playground ./week-1/day-6/env-playground

Inject variables dynamically during boot. Pass runtime parameters using the explicit environment CLI parameters (-e) to override build-time values:

$ docker run -d --name env-app -e WELCOME_MESSAGE="Overrides activated! ⚡" -e THEME_COLOR="#a855f7" -p 8080:3000 env-playground

Bulk load configurations. Perfect for production variables and credentials. Maintain keys in a local .env file and inject all of them securely in a single CLI argument:

$ docker run -d --name env-app --env-file ./week-1/day-6/env-playground/.env -p 8080:3000 env-playground

⚠️ Environment Variables FAQ

Q1: Why is it highly dangerous to commit .env files to GitHub?
Local .env files contain raw, unencrypted database credentials, API access secrets, and security hashes. Committing them lets automated bots scrape your repository in seconds, leading to security breaches. Always list your .env in your project's .gitignore!
Q2: What is the main difference between ARG and ENV instructions?
**ARG** defines build-time variables (only accessible during docker build). **ENV** defines runtime environment variables (injected into the container shell and preserved in container memory while running).
Q3: Do I need to rebuild my container image if I change variables inside my local .env file?
**No.** This is the core magic of Environment Variables! You only need to stop, remove, and spin up a new container instance linked to the updated .env file. The compiled image filesystem remains completely unchanged.
